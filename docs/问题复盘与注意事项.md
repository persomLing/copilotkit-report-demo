# CopilotKit 研报工作台 Demo 问题复盘与注意事项

本文记录本次 React AI 化 demo 开发过程中遇到的问题、修复方式，以及迁移到真实金融系统时需要提前设计的约束。

## 1. 技术选型问题

最初讨论过几条路线：

- CopilotKit：适合旧 React 项目渐进改造，前端通过 `useAgentContext` 暴露状态，通过 `useFrontendTool` 注册动作。
- Vercel AI SDK：更适合 Next.js App Router 和 RSC 场景，对普通 Vite/CRA 旧项目不够自然。
- 自研 ActionExecutor：适合页面少、动作简单、团队想完全掌控协议的情况。
- Tambo/GenUI：偏动态生成 UI，不适合“控制已有业务页面”的中后台改造。

本 demo 最终选择 CopilotKit，是因为它更贴近“AI 控制已有页面状态和业务动作”的目标。

## 2. 后端 Runtime 没启动导致前端不响应

现象：

- Copilot 对话输入后没有继续执行。
- Vite 控制台出现 `http proxy error: /api/copilotkit` 和 `ECONNREFUSED`。

原因：

- 前端 `5174` 正常运行，但 CopilotKit Runtime 后端 `4000` 没有监听。
- 前端通过 Vite proxy 把 `/api/copilotkit` 转发到 `http://localhost:4000`，目标不存在就会拒绝连接。

处理：

- 单独启动 `npm run dev:server`。
- 用 `http://localhost:4000/health` 检查后端是否正常。

注意：

- 开发环境要同时启动前端和 Runtime。
- Docker 版已改成 Express 同时托管前端静态资源和 `/api/copilotkit`，只暴露 `4000`。

## 3. DeepSeek 模型接入问题

遇到过 `Unsupported model version v4 for provider "openai.responses"`。

原因：

- DeepSeek 不能按 OpenAI Responses provider 的模型版本理解。
- 应使用 AI SDK 的 OpenAI-compatible chat provider，并配置 DeepSeek `baseURL`。

当前实现：

```js
const deepseek = createOpenAI({
  apiKey: process.env.DEEPSEEK_API_KEY,
  baseURL: "https://api.deepseek.com",
});

return deepseek.chat("deepseek-v4-flash");
```

注意：

- API Key 只放后端 `.env`，不要进入前端环境变量。
- 不要把真实 `.env` 打进 Docker 镜像或提交仓库。

## 4. CopilotKit 默认 UI 占用空间

现象：

- 默认聊天 UI 会占用页面右侧较大区域。

处理：

- 改成悬浮按钮触发 `CopilotPopup`。
- 只有用户点击按钮后才挂载对话窗口，主页面保持工作台布局。

注意：

- 旧项目改造时，AI 入口应尽量不破坏原业务页面布局。
- 中后台系统更适合悬浮按钮、抽屉或右下角弹窗，而不是强占一整列。

## 5. AI 连续工具调用读到旧 React state

现象：

- AI 先调用 `selectReports`，再调用 `exportSelectedReports`。
- 第一个工具显示已选中，但第二个工具仍提示“没有选中的研报”。

原因：

- React `setState` 是异步的。
- 连续工具调用中，后一个工具读到的是旧闭包里的 `selectedIds`。

处理：

- 对关键状态增加同步 ref：
  - `reportsRef`
  - `selectedIdsRef`
  - `filterRef`
  - `secondaryOptionsRef`
- 工具 handler 读 ref，UI 渲染读 state。

注意：

- Copilot 工具 handler 不能只依赖 React state 闭包。
- 所有 AI 动作需要一个“最新事实源”，尤其是列表、选中项、当前筛选条件。

## 6. “导出当前页”和“导出选中项”语义混淆

现象：

- 用户说“导出当前页面的数据”，AI 先调用 `selectReports` 再调用 `exportSelectedReports`。
- 这条链路复杂且容易受状态时序影响。

处理：

- 新增 `exportVisibleReports`，直接导出当前可见表格。
- `exportSelectedReports` 只处理已选中项。
- 手动导出和 AI 导出复用同一套 CSV 生成逻辑。

注意：

- 一个用户意图最好对应一个清晰工具。
- 不要让 AI 为了完成简单动作绕多步，除非确实需要审批或前置状态。

## 7. 二级筛选不能直接暴露给 AI

业务要求：

- 二级筛选数据应来自接口请求。
- AI 不应在不知道真实二级候选项时直接设置二级筛选。

处理：

- 页面上下文不再直接暴露 `secondaryOptions` 数组。
- 新增 `resolveSecondaryFilterIntent`。
- 该工具只返回金融语义提示和一级方向，不返回可执行二级确认。
- 用户确认后才允许调用 `loadSecondaryFilterOptions`。

注意：

- 二级筛选属于依赖型槽位。
- 用户直接说“可转债”“新能源”“主动权益”时，AI 应先识别和追问，而不是直接填值。

## 8. 二级筛选需要用户确认

处理：

- `setReportFilter` 增加 `secondaryConfirmedByUser` 参数。
- 如果 AI 设置具体 `secondaryCategory` 但没有传 `secondaryConfirmedByUser=true`，工具会拒绝执行。

意义：

- 这不是只靠 prompt，而是工具层硬约束。
- 即使模型误判，也不能绕过确认流程。

## 9. 接口耗时、失败和重试

业务要求：

- AI 调用接口时应等待。
- 接口请求失败后应自动重试一次。
- 如果仍失败，应告知用户，不再继续执行后续筛选。

处理：

- 新增 `requestSecondaryOptions` 模拟接口耗时。
- `fetchSecondaryOptions` 会等待接口返回。
- 第一次失败自动重试。
- 第二次仍失败返回 `ok=false`。
- `loadSecondaryFilterOptions` 将 `ok/attempts/error/nextStep` 返回给 AI。

注意：

- 真实项目里应把重试策略放在 service 层，而不是散落在 UI 组件里。
- 工具返回值要结构化，让 AI 明确知道是否能继续下一步。

## 10. 敏感动作需要人工确认

导出、删除、提交、批量修改等动作都应视为敏感动作。

当前 demo：

- `exportSelectedReports` 会弹 `window.confirm`。
- `exportVisibleReports` 当前直接导出当前页，真实项目中也可按数据敏感级别加审批。

注意：

- 金融系统里，AI 不应直接执行不可逆或合规敏感动作。
- 建议所有敏感工具统一支持 `renderAndWait` 或前端确认弹窗。

## 11. 旧项目改造建议

真实项目改造时建议按顺序做：

1. 先选一个高价值页面，例如研报列表、基金筛选器、客户查询。
2. 梳理页面状态、接口、用户动作。
3. 把分散状态收敛到 store 或局部状态管理层。
4. 只暴露稳定、可审计的 action 给 AI。
5. 对依赖型筛选、敏感动作、接口失败做硬约束。
6. 再逐步扩大到更多页面。

## 12. 需要长期注意的点

- 不要把模型 API Key 放前端。
- 不要让 AI 直接操作 DOM 作为第一选择，除非项目状态完全不可控。
- 工具名称、prompt 描述和实际 handler 要严格一致。
- 工具参数要用枚举和 schema 限制。
- 工具结果要包含是否成功、错误原因和下一步建议。
- 页面上下文只暴露 AI 必须知道的信息，不要暴露不该提前知道的接口数据。
- AI 动作必须可审计，至少记录动作名、参数、结果和时间。
