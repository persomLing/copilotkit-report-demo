### 15. AG-UI 软打断与共享操作日志

本次根据业务语义重新定义“打断”：

- 用户打断只停止 AI 当前 run 的继续输出和后续工具规划。
- 已经发出的前端业务请求不再取消。
- 二级筛选接口、研报加载等请求完成后仍然写入共享页面状态。
- 这样更接近真实 B 端系统：进入后端/接口执行阶段的动作默认继续完成，避免半路取消导致业务状态不确定。

实现调整：

- 保留 `agent.abortRun()`。
- 移除前端业务请求对 `AbortController` 的依赖。
- 将运行中提示从“请求可取消”调整为“AI 后续操作可打断，已发请求继续完成”。
- 后端 agent prompt 同步更新，避免 AI 误以为业务请求被取消。

### 20. 基金主题词与行业二级词歧义优化

- 前端 `src/App.jsx`

### 21. 追问策略改为置信度决策

问题：

- 仅靠“疑似二级词默认追问”会让 AI 像表单校验器，用户容易觉得不如手动筛选。
- 对筛选、排序、关键词检索这类可撤销动作，应该尽量先做，再允许用户纠正。

策略调整：

- 高置信度：直接执行。
- 中置信度：先执行，并说明当前假设。
- 低置信度：才追问。
- 敏感动作，例如导出、删除、提交，仍然必须确认。

实现：

- `src/reportStore.js`
  - 新增 `resolveReportFilterIntent(userText)`。
  - 返回 `confidence`、`action`、`filterPatch`、`notes`、`nextSteps`。
  - 修复中文子串误判：`评分不低于80` 不再把“低”识别为风险等级，只有“低风险/风险低”才映射 `risk=低`。
  - `评分不低于80` 只映射 `scoreMin=80`，不自动触发按评分排序。
- `src/App.jsx`
  - 新增 CopilotKit 前端工具 `resolveReportFilterIntent`。
  - 在 `useAgentContext` 中要求筛选类自然语言优先调用该工具，再根据 `action` 决定执行或追问。
- `server/index.js`
  - 后端 Agent prompt 改为优先使用整体意图解析工具。
  - 明确不要因为出现疑似二级词就默认追问。

### 22. 新增用户记忆与系统规则沉淀

背景：

- 只靠固定 prompt 无法持续吸收开发过程中的经验。
- 用户反馈里有两类信息需要分开处理：
  - 当前用户自己的偏好，例如“别一直追问，能执行就先执行”。
  - 可复用的系统规则，例如“半导体相关基金应走基金研究 + keyword，不应走股票二级行业”。
- 如果把所有反馈都直接写成全局规则，会污染系统；如果完全不沉淀，AI 会反复犯同类错误。

设计：

- 新增两层知识体系：
  - `userMemory`：当前用户偏好、习惯、纠错。
  - `systemRules`：包含 `approved` 已生效规则、`candidates` 待审核规则、`caseLog` 学习案例。
- 规则优先级：
  1. 前端工具返回值和业务接口结果。
  2. 已批准系统规则。
  3. 当前用户记忆。
  4. 当前对话上下文。
  5. 模型常识。
- 待审核规则只能展示和等待人工批准，不能直接当事实执行。

实现：

- `src/memoryStore.js`
  - 新增 localStorage 持久化。
  - 新增默认规则：评分阈值、主题基金、明确一级二级时不追问。
  - 新增 `extractRuleCandidateFromFeedback`，把一次纠错启发式提炼成候选规则。
- `src/App.jsx`
  - 新增 `knowledgeBase` 状态。
  - 通过 `useAgentContext` 暴露 `learningMemory`。
  - 新增 CopilotKit 工具：
    - `recordUserMemory`
    - `recordLearningCase`
    - `proposeSystemRule`
    - `approveSystemRule`
    - `dismissSystemRule`
    - `forgetUserMemory`
  - 新增“知识沉淀”面板，展示用户记忆、已生效规则、待审核规则。
- `server/index.js`
  - Agent prompt 增加学习体系说明。
  - 明确 `pendingRuleCandidates` 不能直接执行。
- `README.md`
  - 补充学习体系说明和技术设计入口。
- `docs/技术设计.md`
  - 新增整体架构、工具链路、打断策略、两层学习体系说明。

注意：

- 当前 demo 使用 localStorage，生产环境应迁移到后端存储。
- 批准系统规则应接权限、审计和灰度发布。
- 规则抽取不应完全交给模型自动生效，至少要经过人工审核或离线评估。

### 23. 补充多模块多筛选风格设计

背景：

- 当前 demo 只有研报筛选一个模块。
- 真实系统中可能同时存在研报、基金、基金经理、组合、产品等多个模块。
- 每个模块的筛选字段、业务语义、接口链路都不同，不能让大模型靠常识自由猜。

补充设计：

- 在 `docs/技术设计.md` 新增“多模块多筛选风格扩展”。
- 明确模型区分模块依赖四类信息：
  - 当前页面状态。
  - 模块语义描述。
  - 模块筛选 schema。
  - 模块独立工具。
- 推荐 `AI Shell + Module Adapter` 架构：
  - `ReportModule`
  - `FundModule`
  - `ManagerModule`
  - `PortfolioModule`
- 推荐两段式上下文：
  1. 先暴露模块目录。
  2. 通过 `resolveModuleIntent(userText)` 选择模块。
  3. 再暴露选中模块的详细 schema、状态和工具。
- 多模块规则优先级：

```text
工具/接口事实 > 全局规则 > 当前模块规则 > 用户记忆 > 对话上下文 > 模型常识
```

结论：

- 不要做一个超大的 `setAnyFilter`。
- 模块越多，越要把 state、schema、tools、workflowRules 分模块注册。
- 多模块场景下，AI 的关键能力不是“猜得更准”，而是被清晰的模块边界和工具约束住。

### 24. 规则库改为按需召回，避免全量规则进入上下文

背景：

- 规则越来越多时，如果每轮都把全量 `approvedSystemRules` 放进上下文，会浪费 token。
- 全量规则也会增加模型匹配成本，并提升规则冲突概率。
- 生产系统应该把规则库当成可检索知识库，而不是 prompt 附录。

实现：

- `src/memoryStore.js`
  - 规则结构增加 `scope`、`moduleId`、`priority`、`summary`、`triggers`。
  - 新增 `retrieveRelevantRules(knowledgeBase, { query, moduleId, topK })`。
  - `summarizeKnowledgeForAgent` 不再返回全量 `approvedSystemRules`，只返回 `relevantRules` 和规则数量统计。

### 25. 限制 Copilot 对话区复刻业务表格

### 26. 用户指正后自动补充待审核规则并重新处理

问题：

- 用户指正“以后遇到类似情况应该优先看二级分类，没有匹配再用关键词”后，AI 只调用了 `recordUserMemory`。
- 这会进入“用户记忆”，但不会进入“待审核规则”。
- AI 也没有按用户纠正后的逻辑重新处理当前筛选任务。

原因：

- 原设计中：
  - `recordUserMemory` 只保存当前用户偏好。
  - `recordLearningCase` 或 `proposeSystemRule` 才会生成待审核规则。
- 模型选择了“记忆工具”，没有继续调用“规则候选工具”和业务筛选工具。

调整：

- `src/App.jsx`
  - `recordUserMemory` 增加兜底判断。
  - 如果记忆内容包含“以后、应该、优先、不要、遇到类似情况”等可复用表达，会同步生成一条待审核规则候选。
  - 工具返回值提示：如果纠正的是当前筛选，还应继续重新执行当前任务。
- `server/index.js`
  - Agent prompt 明确：用户纠正当前筛选/排序/模块判断时，不要只道歉或只记录记忆；应按纠正后的逻辑重新调用工具处理当前任务。
  - 可复用表达应调用 `recordLearningCase` 或 `proposeSystemRule` 生成待审核规则。

结果：

- 单纯个人偏好仍进入用户记忆。
- 可复用业务判断会进入待审核规则。
- 当前任务的可执行纠正会触发重新处理，而不是停在解释。

### 27. 修复 AI 切换一级分类后二级候选为空

问题：

- 手动切换一级分类时会调用 `fetchSecondaryOptions`，所以二级下拉能正常加载。
- AI 有时只调用 `setReportFilter({ primaryCategory })`，旧逻辑只更新一级筛选，没有自动拉取该一级下的二级候选项。
- 结果是一级已经切换，但二级分类下拉没有对应数据。

修复：

- `src/App.jsx`
  - `setReportFilter` 检测到 `primaryCategory` 变化时，自动调用 `fetchSecondaryOptions`。
  - 如果二级候选加载失败，停止本次筛选设置，避免页面进入半更新状态。
  - 如果只切换一级、不指定具体二级，则自动把 `secondaryCategory` 重置为 `全部`。
- `server/index.js`
  - Agent prompt 补充：只切换一级分类时可以调用 `setReportFilter`，前端会自动加载二级候选项。

保留约束：

- 如果 AI 要设置具体二级筛选，仍然必须确保用户已确认，并且该二级值存在于接口返回的候选项中。
### 28. 硬化 Copilot 对话区禁止复刻业务表格

问题：
- 只在 prompt 中要求“不要输出 Markdown 表格”属于软约束，模型仍可能根据上下文里的完整 `visibleReports` 字段自行组织表格。
- 当前页研报的券商、评级、评分、热度、日期等字段已经在左侧业务表格展示，继续暴露给 AI 会增加 token，也会诱导模型在聊天区复刻表格。

调整：
- `src/App.jsx`
  - 将 AI 上下文中的 `visibleReports` 改为 `visibleReportRefs`。
  - `visibleReportRefs` 只保留 `position/id/title`，用于打开详情、选中、导出等定位操作。
  - 新增 `visibleReportContextPolicy`，明确完整字段以左侧业务表格为准。
  - `responsePolicy` 改为禁止复刻当前页数据，而不是只禁止复刻某个字段名。
- `server/index.js`
  - Agent prompt 同步改为 `visibleReportRefs`。
  - 明确前端不会再提供完整表格字段，筛选/排序/加载后只返回简短摘要。
- `src/styles.css`
  - 增加 Copilot 对话区 Markdown 表格隐藏兜底，防止旧会话或偶发表格输出继续影响界面。

结论：
- 业务表格由工作台负责展示。
- Copilot 只负责说明动作结果、异常、下一步建议，以及通过工具操作页面。
- 对话区不再作为第二个表格渲染区域。
